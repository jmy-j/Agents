## 1. 系统概览

本系统是一个基于 **RAG（Retrieval-Augmented Generation，检索增强生成）** 架构的私有文档问答工具。核心能力：**上传 .docx 文档后，用自然语言对文档内容进行提问，系统会从文档中找到相关内容并生成准确回答。**

### 技术栈

| 组件 | 技术 | 作用 |
|------|------|------|
| 文档解析 | `python-docx` | 提取 .docx 文件中的文字 |
| 文本分块 | LangChain `RecursiveCharacterTextSplitter` | 将长文档切成小段 |
| 文本向量化 | Ollama + `nomic-embed-text` | 把文字转成数学向量 |
| 向量存储 | ChromaDB | 存储向量 + 原文，支持相似度检索 |
| 对话模型 | Ollama + `qwen2.5:7b` | 根据检索结果生成自然语言回答 |
| Web 界面 | Streamlit | 浏览器中的聊天式交互界面 |

### 为什么是 RAG 而不是直接问 LLM？

- **LLM 有知识截止日期**，不知道你文档里的内容
- **LLM 可能编造答案**（幻觉），RAG 要求它严格依据文档回答
- **数据隐私**：所有数据存在本地，不经过任何外部 API

## 2. 环境准备

### 2.1 安装 Ollama

1. 访问 [ollama.com](https://ollama.com) 下载 Windows 版安装程序
2. 安装完成后，Ollama 会在后台运行（任务栏可见图标）

### 2.2 下载所需模型

打开 PowerShell 终端，依次执行：

```powershell
# Embedding 模型（把文字变成向量，274MB）
ollama pull nomic-embed-text

# 对话模型（生成回答，4.7GB，如果网速慢可换 qwen2.5:1.5b）
ollama pull qwen2.5:7b
```

验证模型已就绪：

```powershell
ollama list
# 应看到 nomic-embed-text 和 qwen2.5:7b
```

### 2.3 安装 Python 依赖

```powershell
cd D:\python-project\py-project-1\RAG
pip install -r requirements.txt
```

---

## 3. 启动系统

```powershell
cd D:\python-project\py-project-1\RAG
streamlit run app.py
```

浏览器会自动打开 `http://localhost:8501`，看到聊天界面即启动成功。

> 确保 Ollama 在后台运行（任务栏有图标），否则会报连接错误。

### 4.1 界面布局

│  └──────────────────┘       │   └──────────────────────┘   │

### 4.2 上传文档

**方式一：单文件上传**
1. 在侧边栏点击 "Browse files"
2. 选择一个 .docx 文件
3. 系统自动解析并显示 "已添加 N 个文本块到向量库"

**方式二：批量导入**
1. 将所有 .docx 文件放在同一个文件夹中（如 `D:\docs\`）
2. 在侧边栏输入文件夹路径
3. 点击 "导入文件夹"

### 4.3 提问

在底部输入框输入问题，例如：

- "这份文档的主要内容是什么？"
- "第二章讲了什么？"
- "文档中提到的主要人物有哪些？"
- "关于 XXX 项目，文档里是怎么说的？"

### 4.4 使用技巧

- **问题越具体，回答越精确**。不要问 "概括一下"，而是问 "文档中关于项目实施时间表的部分是怎么说的？"
- **可以连续追问**。系统会记住对话历史（但每次检索独立进行）
- **向量库是持久化的**。关闭浏览器后数据不丢失，下次启动直接可用
- **不同文档的相同主题**：上传多份文档到一个向量库，跨文档提问也能找到答案

## 5. 数据流转全貌

### 5.1 总览

整个系统有两条数据流：**入库流（文档 → 向量库）**和**查询流（问题 → 回答）**。

### 5.2 入库流：文档 → 向量库

这是上传 .docx 文件时发生的完整流程。

```
                    ┌──────────────┐
                    │   .docx 文件  │
                    │  (原始文档)   │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Step 1      │
                    │  解析文档     │  python-docx 逐段提取文字
                    │              │  过滤空段落，拼接成完整文本
                    └──────┬───────┘
                           │  纯文本字符串
                    ┌──────▼───────┐
                    │  Step 2      │
                    │  文本分块     │  RecursiveCharacterTextSplitter
                    │              │  chunk_size=500, overlap=50
                    │              │  按段落→句子→空格→字符优先级切割
                    └──────┬───────┘
                           │  N 个文本块 (每块 ~500 字)
                    ┌──────▼───────┐
                    │  Step 3      │
                    │  向量化       │  Ollama + nomic-embed-text
                    │              │  每个文本块 → 768维向量
                    └──────┬───────┘
                           │  N 个向量 + N 段原文
                    ┌──────▼───────┐
                    │  Step 4      │
                    │  存入向量库   │  ChromaDB 持久化到磁盘
                    │              │  ./chroma_db/ 目录
                    └──────────────┘
```

**关键细节：**
- 分块时相邻块有 50 字重叠，防止关键信息正好落在边界被切断
- 每个块都保留元数据（来源文件名、块序号），回答时可以溯源
- ChromaDB 数据存在 `./chroma_db/` 文件夹，删除即清空向量库

### 5.3 查询流：问题 → 回答

这是用户在对话框提问时发生的完整流程。

```
                    ┌──────────────┐
                    │  用户提问     │
                    │  "文档里提到  │
                    │   的截止日期   │
                    │   是什么？"    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Step 1      │
                    │  问题向量化   │  Ollama + nomic-embed-text
                    │              │  问题 → 768维向量
                    └──────┬───────┘
                           │  问题向量
                    ┌──────▼───────┐
                    │  Step 2      │
                    │  相似度检索   │  ChromaDB.similarity_search()
                    │              │  问题向量 vs 库中所有文档向量
                    │              │  返回 top-4 最相似的文本块
                    └──────┬───────┘
                           │  4 个最相关的文本块 + 元数据
                    ┌──────▼───────┐
                    │  Step 3      │
                    │  拼接 Prompt  │  构建结构化提示词：
                    │              │  ┌─────────────────────┐
                    │              │  │ 系统指令              │
                    │              │  │ 文档内容（4个检索块）  │
                    │              │  │ 用户问题              │
                    │              │  │ 回答要求（依据文档）   │
                    │              │  └─────────────────────┘
                    └──────┬───────┘
                           │  完整 Prompt 文本
                    ┌──────▼───────┐
                    │  Step 4      │
                    │  LLM 生成     │  Ollama + qwen2.5:7b
                    │              │  依据 Prompt 生成自然语言回答
                    │              │  如果文档无相关信息，如实说明
                    └──────┬───────┘
                           │  回答文本
                    ┌──────▼───────┐
                    │  返回前端     │  在浏览器对话气泡中显示
                    │              │  同时保留到聊天历史
                    └──────────────┘
```

## 6. 配置参考

配置文件在 `rag_core.py` 顶部：

```python
EMBEDDING_MODEL = "nomic-embed-text"   # embedding 模型名
CHAT_MODEL = "qwen2.5:7b"              # 对话模型名（可改为 qwen2.5:1.5b 等）
CHUNK_SIZE = 500                       # 每块字数（越小检索越精确但可能丢失上下文）
CHUNK_OVERLAP = 50                     # 块间重叠字数（建议设为 CHUNK_SIZE 的 10%）
SEARCH_K = 4                           # 每次检索返回的块数
CHROMA_DIR = "./chroma_db"             # 向量库存储路径
```

### 调参建议

| 场景 | CHUNK_SIZE | CHUNK_OVERLAP | SEARCH_K |
|------|------------|----------------|----------|
| 短文档（< 10页） | 300 | 30 | 3 |
| 中等文档（10-50页） | 500 | 50 | 4 |
| 长文档（> 50页） | 800 | 80 | 6 |
| 精确查询（找具体数据） | 200 | 20 | 5 |
| 概览性提问（总结、概述） | 800 | 100 | 3 |

## 7. 常见问题

### Q: 启动时报错 "Connection refused"？

A: 确认 Ollama 在后台运行（任务栏有图标）。如果没有，手动启动 Ollama 应用。

### Q: 上传 .docx 后提示 "文档为空"？

A: 检查该 .docx 是否确实包含文字（而不是纯图片或扫描件）。Word 中的文本框、艺术字可能无法被 python-docx 提取。

### Q: 回答与文档内容无关？在胡说八道？

A: 可能原因及解决方案：
1. 提问太笼统 → 把问题写得更具体
2. 检索块数不够 → 调大 `SEARCH_K`
3. 文本块太大粒度不够 → 调小 `CHUNK_SIZE`
4. 对话模型能力不足 → 换更大的模型（如 `qwen2.5:14b`）

### Q: 如何清空向量库？

A: 删除 `rag_core.py` 同级目录下的 `chroma_db/` 文件夹，重新启动应用即可。

### Q: 可以同时索引多个文档吗？

A: 可以。上传多个文件后，所有文档的文本块都在同一个向量库中，提问时会跨文档检索。

### Q: 文档内容会泄露到外网吗？

A: 不会。Ollama 和 ChromaDB 都是纯本地运行，所有数据（文档、向量、模型推理）都在你的电脑上完成，不需要网络连接（模型下载除外）。

### Q: 旧版 .doc 格式支持吗？

A: 当前只支持 .docx。.doc 文件可以用 Word 打开后另存为 .docx，或用 LibreOffice 批量转换。
